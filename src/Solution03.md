# Solution03: Atomic 타입(CAS 알고리즘), ConcurrentHashMap vs HashMap

## 1. 초심자용 비유로 이해하기

### ❶ `AtomicInteger`와 CAS (Compare-And-Swap)

* **비유**: **"물어보고 잽싸게 바꾸기"**
* **상세**: 기존의 동기화 방식(`synchronized`)은 자원에 자물쇠를 걸어 잠그는 방식(비관적 락)입니다. 반면 `AtomicInteger`는 자물쇠를 걸지 않고 다음과 같은 **CAS(비교 후 교체)
  ** 방식으로 작동합니다.
    1. 값을 바꾸고 싶을 때, 먼저 현재 값(예: `10`)을 확인합니다.
    2. 새로운 값(`11`)을 들고 가서 컴퓨터에게 묻습니다. **"내가 처음에 봤을 때 값이 10이었는데, 지금도 여전히 10이니? 그렇다면 11로 바꿔줘."**
    3. 맞다면 11로 바로 바꿉니다. 만약 그 사이에 다른 스레드가 먼저 10을 12로 바꿨다면 실패하고, 다시 처음부터 시도(루프 돌며 재시도)합니다.
* 자물쇠를 거는 오버헤드가 없기 때문에 스레드 간의 락 획득 대기가 생기지 않아 성능이 더 좋습니다.

### ❷ `ConcurrentHashMap` vs `HashMap`

* **비유**: **"1인용 화장실과 칸이 나누어진 공중 화장실"**
* **상세**:
    * `HashMap`은 스레드 안전 장치가 전혀 없어서, 동시에 접근하면 데이터가 유실되거나 내부 자료구조가 깨져 무한 루프가 발생할 수 있습니다.
    * 그렇다고 모든 메서드에 `synchronized`를 걸어 둔 옛날의 `Hashtable`이나 `Collections.synchronizedMap()`은 맵 전체에 자물쇠를 거는 **1인용 화장실**과
      같습니다. 100명이 와도 1명만 들어가고 99명은 줄을 서서 기다려야 합니다.
    * `ConcurrentHashMap`은 맵을 여러 구역(버킷)으로 쪼개어, 각 구역마다 개별 자물쇠를 단 **여러 칸짜리 공중 화장실**과 같습니다. 철수가 1번 칸(1번 버킷)을 쓰고 있어도 영희는 2번
      칸(2번 버킷)을 동시에 쓸 수 있습니다. 따라서 멀티스레드 환경에서도 성능 저하를 최소화하면서 안전하게 데이터를 삽입/삭제할 수 있습니다.

## 2. 예상 면접 질문 & 모범 답변

### Q1. CAS(Compare-And-Swap) 알고리즘이란 무엇이며, Atomic 클래스들이 이를 어떻게 활용하나요?

* **답변**: CAS는 멀티스레드 환경에서 동기화 없이 값을 안전하게 변경하기 위한 CPU 하드웨어 수준의 명령어입니다.
    * 메모리 위치의 현재 값(Expected Value)이 내가 알고 있는 값과 일치하는지 비교하고, 일치할 때만 새로운 값(New Value)으로 변경(Swap)합니다.
    * `AtomicInteger`나 `AtomicLong` 등은 내부적으로 루프를 돌며 이 CAS 연산이 성공할 때까지 재시도하는 방식으로 구현되어 있어, 무거운 `synchronized` 락 없이도 원자성을
      보장합니다. 이를 락-프리(Lock-Free) 혹은 낙관적 동기화 방식이라고 합니다.

### Q2. `HashMap`을 멀티스레드 환경에서 그대로 사용하면 어떤 문제가 발생하나요?

* **답변**: `HashMap`은 Thread-safe하지 않은 컬렉션입니다.
    * 여러 스레드가 동시에 `put()` 연산을 수행하여 같은 버킷에 값을 쓰거나 해시 충돌(Hash Collision)을 처리하는 과정에서, 이진 트리나 링크드 리스트의 구조가 깨져 데이터가 유실될 수
      있습니다.
    * 최악의 경우, JDK 7 이하 버전에선 내부 링크가 순환 참조 구조를 형성하면서 CPU 점유율이 100%에 달하는 무한 루프(Infinite Loop) 현상이 발생하기도 했습니다. 따라서 멀티스레드
      환경에서는 절대 일반 `HashMap`을 사용해서는 안 됩니다.

### Q3. `ConcurrentHashMap`은 어떻게 성능 향상과 스레드 안전(Thread-safety)을 동시에 달성하나요?

* **답변**: `ConcurrentHashMap`은 맵 전체를 잠그는 대신, 데이터를 버킷 단위로 세분화하여 잠그는 **락 스트라이핑(Lock Striping)** 기법을 사용합니다.
    * 내부적으로 쓰기 작업(`put()`) 시 전체 맵을 잠그지 않고 해당 버킷(노드)에만 `synchronized` 블록을 씌우거나 CAS 연산을 사용하므로, 서로 다른 버킷에 접근하는 스레드들은 락 경쟁
      없이 병렬로 동작합니다.
    * 또한, 읽기 작업(`get()`)은 기본적으로 락을 걸지 않고 `volatile` 키워드를 통해 최신 값을 읽도록 설계되어 읽기 성능이 매우 뛰어납니다.
