# Solution04: 스레드 풀 (Thread Pool) & ExecutorService

## 1. 초심자용 비유로 이해하기

### ❶ 스레드 풀 (Thread Pool)

* **비유**: **"자체 택시를 운용하는 콜택시 회사"**
* **상세**: 손님(작업)이 택시를 요청할 때마다 새 차(스레드)를 사고, 목적지에 도착하자마자 그 차를 폐차장에 보내 부수는 회사가 있다고 해봅시다. 차를 새로 사고 폐차하는 비용(스레드 생성 및 소멸 비용)이
  어마어마하게 들 것입니다. 또한 동시에 손님 10,000명이 몰리면 길거리에 차가 10,000대나 돌아다녀 도로가 마비(CPU 컨텍스트 스위칭 과부하)됩니다.
* 대신 **스레드 풀**은 택시 4대(스레드 개수 설정)를 미리 준비해 두고 계속해서 재사용합니다.
    1. 손님이 오면 대기 중인 택시에 태워 보냅니다.
    2. 만약 모든 택시가 운행 중이면, 뒤이어 온 손님들은 대기실(작업 큐, Work Queue)에서 줄을 서서 기다립니다.
    3. 운행을 마친 택시는 차고지로 돌아와 대기실의 다음 손님을 태웁니다.
* 이 방식을 통해 스레드 생성 오버헤드를 막고, 시스템 리소스를 안정적으로 제어할 수 있습니다.

## 2. 예상 면접 질문 & 모범 답변

### Q1. 매번 `new Thread()`를 호출하여 스레드를 생성하는 대신 스레드 풀(Thread Pool)을 사용하는 이유는 무엇인가요?

* **답변**:
    1. **생성 및 소멸 비용 절감**: 자바 스레드는 OS 스레드와 일대일로 매핑되는 무거운 자원입니다. 스레드를 미리 만들어 두고 재사용함으로써 스레드 생성/소멸에 필요한 시스템 리소스 비용을 크게 아낄 수
       있습니다.
    2. **응답성 성능 향상**: 작업이 요청되었을 때 스레드가 이미 준비되어 있으므로, 스레드 생성 단계를 거치지 않고 즉시 작업을 실행할 수 있어 응답 시간이 줄어듭니다.
    3. **리소스 무제한 증가 방지**: 과도한 스레드 생성으로 인한 메모리 고갈(OOM) 및 극심한 CPU 컨텍스트 스위칭 오버헤드를 방지하고, 동시 처리 가능한 작업 개수를 일정 수준으로 제어하여 시스템
       안정성을 보장합니다.

### Q2. `ExecutorService`에서 `shutdown()`과 `shutdownNow()`의 차이점은 무엇인가요?

* **답변**:
    * **`shutdown()`**: 새로운 작업 제출(Submit)은 거부하지만, 이미 작업 큐(Queue)에 들어와 있는 작업들과 현재 실행 중인 작업들은 끝까지 정상적으로 완료한 후 스레드 풀을
      종료합니다. (Graceful Shutdown)
    * **`shutdownNow()`**: 현재 실행 중인 작업들을 강제로 인터럽트(interrupt)하여 중단시키고, 대기 중이던 작업 큐의 미실행 작업 목록을 반환한 후 즉시 스레드 풀을 종료합니다.

### Q3. 스레드 풀의 적절한 스레드 개수(Core Pool Size)는 어떻게 결정하나요?

* **답변**: 작업의 성격(CPU 바운드 작업인지, I/O 바운드 작업인지)에 따라 다르게 설정해야 합니다.
    * **CPU 바운드 작업 (연산 중심)**: CPU 코어를 최대한 활용해야 하므로 대기 시간이 적습니다. 컨텍스트 스위칭 비용을 최소화하기 위해 **`CPU 코어 수 + 1`** 정도로 스레드를 낮게
      설정하는 것이 좋습니다.
    * **I/O 바운드 작업 (DB 조회, 네트워크 통신 중심)**: 데이터 요청 후 응답이 올 때까지 스레드가 대기(Block)하는 시간이 깁니다. 따라서 스레드를 많이 만들어도 CPU가 놀기 때문에 *
      *`CPU 코어 수 * 2`** 또는 그 이상의 넉넉한 스레드를 할당하여 병렬 처리 효율을 높이는 것이 적절합니다.
